实测Claude Sonnet 4.6百万上下文：我扔了一整个项目进去，结果...

发布于 2026年3月12日 · AI编程 · Claude

说实话，我一开始是拒绝用Claude的。

作为一个从GPT-3.5时代就开始用AI辅助编程的人，我已经习惯了"分段投喂"的工作流——把代码切成一小块一小块，小心翼翼地控制上下文长度，生怕超出限制。这种体验，怎么说呢，就像是在用一个只有7秒记忆的金鱼当助手。

直到上周，Anthropic发布了Claude Sonnet 4.6。

百万Token上下文窗口。什么概念？

  • 整个Linux内核代码库？能装下。
  • 一本《战争与和平》？能装下。
  • 我那个祖传屎山项目？绰绰有余。

我决定做个实验：把我手头一个3万行代码的中型项目直接扔进去，看看会发生什么。

实验准备：拿到API Key的曲折路

先说前提，想测试Sonnet 4.6，你得先能调用它。

Anthropic的API在国内访问不太友好，直接请求经常超时或者干脆连不上。而且他们的API Key申请还有点门槛，新账号经常遇到各种限制。

我折腾了半天，最后决定用一个国内的中转API服务——KL-API。这个是我之前做项目时用过的，主要优势是：

  • 国内节点，延迟低
  • 支持Claude全系列模型，包括最新的Sonnet 4.6
  • 价格透明，按官方汇率结算
  • 不需要特殊网络环境

配置也很简单，就改一下base URL和api key：

import openai

# 原来是直接调Anthropic
# client = anthropic.Anthropic(api_key="sk-ant-xxxxx")

# 改用KL-API中转
client = openai.OpenAI(
    api_key="sk-你的KL-API-Key",
    base_url="https://api.klai.top/v1"  # KL-API的中转地址
)

# 然后就可以正常调用了
response = client.chat.completions.create(
    model="claude-sonnet-4.6",
    messages=[
        {"role": "user", "content": "帮我分析这段代码..."}
    ],
    max_tokens=8192
)

关于KL-API的注册和使用，有兴趣的可以自行搜索，这里就不展开广告了。主要是解决了我的访问问题，继续往下说实验。

实验一：整项目代码分析

我的测试项目是一个基于FastAPI的后端服务，大概3万多行Python代码，分布在40多个文件里。

以前用GPT-4的时候，我得一个文件一个文件地喂，还要自己记着"这个函数在utils.py里定义了，在service.py里被调用了"。Sonnet 4.6不是号称百万上下文吗？我直接把整个项目的代码打包成一个请求。

具体操作：

# 把整个项目目录的文件内容合并
find . -name "*.py" -type f | head -50 | xargs cat > project_dump.txt
# 结果：大概8万多个Token

然后我发了这个请求：

下面是一个完整的Python项目代码，请帮我：
1. 分析项目的整体架构和模块划分
2. 找出潜在的代码质量问题
3. 识别是否有重复代码可以重构

[此处粘贴8万Token的代码]

等了大概30秒（比我想象的快），Claude的回复让我惊了。

它不仅能准确识别出这是一个FastAPI项目，还详细分析了：

  • 路由层、服务层、数据层的划分
  • 使用了SQLAlchemy作为ORM
  • 采用了依赖注入模式

关键是，它发现了3处我自己都没注意到的代码异味：

  1. 循环导入隐患：models/user.py和schemas/user.py之间有隐式循环依赖
  2. 重复的错误处理逻辑：5个不同的路由文件里都写了几乎一样的try-except块
  3. 未使用的配置项：config.py里定义了3个环境变量，但在代码里根本没用到

最骚的是，它还给出了具体的重构建议，包括代码示例。

关键发现：长上下文不是"能装"，而是"能懂"

以前我以为长上下文就是"能把更多文字塞进去"，这次测试让我明白，真正的价值在于跨文件的关联理解。

Sonnet 4.6能够在一个对话里同时看到：

  • 数据模型的定义（models/）
  • 业务逻辑的实现（services/）
  • API接口的暴露（routers/）
  • 配置和工具函数（config/, utils/）

这种全局视角，让它能发现那种"单看一个文件完全看不出来"的问题。

实验二：批量代码审查

第二个实验更实用：我有一堆PR要审。

我们团队用GitHub，每次发PR都要人工审查。但有些PR改动特别大，涉及十几个文件，几百行变更。这种PR审起来很头疼，经常是"看了后面忘了前面"。

Sonnet 4.6的百万上下文在这里又派上用场了。我可以直接把整个PR的diff打包发给它：

import requests

# 获取PR的diff
pr_diff = requests.get(
    f"https://api.github.com/repos/{owner}/{repo}/pulls/{pr_number}",
    headers={"Accept": "application/vnd.github.v3.diff"}
).text

# 直接发给Claude
review_prompt = f"""
请审查以下Pull Request的代码变更：

{pr_diff}

请从以下维度给出审查意见：
1. 是否有明显的bug或逻辑错误
2. 代码风格是否一致
3. 是否有性能隐患
4. 测试是否充分
5. 总体评价（Approve / Request Changes / Comment）
"""

response = client.chat.completions.create(
    model="claude-sonnet-4.6",
    messages=[{"role": "user", "content": review_prompt}],
    max_tokens=4096
)

我测试了5个真实的PR，结果：

  • 3个PR：Claude发现的问题和人工审查基本一致，还额外指出了2个小问题
  • 1个PR：它发现了一个边界条件处理不当的问题，我们人类审查者都漏掉了
  • 1个PR：误报了一个"问题"，实际上那个设计是刻意的

准确率大概80%左右，作为辅助工具已经很实用了。

实验三：遗留代码理解

最后一个实验是最让我印象深刻的。

我们公司有一个祖传项目，是5年前的前辈写的，用的还是Python 2时代的风格。文档基本没有，注释约等于零，变量命名像是键盘上滚出来的。

以前每次要改这个项目的bug，我都得先花半天时间"考古"，试图理解代码到底在干什么。

这次我把整个项目的代码（大概5万Token）直接发给Sonnet 4.6，问：

这个项目是做什么的？数据流是怎么走的？有哪些核心业务逻辑？

15秒后，我得到了一份比官方文档还清晰的说明：

  • 项目定位：这是一个数据处理管道，负责从多个数据源采集数据，进行清洗转换，然后输出到下游系统
  • 核心流程：采集 → 校验 → 转换 → 分发，四个阶段
  • 关键类：DataSource、Validator、Transformer、Dispatcher
  • 扩展点：如果需要添加新的数据源，应该继承DataSource类并重写fetch方法

它甚至画了一个ASCII流程图（虽然丑但看得懂）。

使用成本：比你想象的便宜

说了这么多优点，肯定有人关心价格。

Claude Sonnet 4.6的定价：

  • 输入：$3 / 百万Token
  • 输出：$15 / 百万Token

百万上下文听起来吓人，但实际使用中有几个省成本的技巧：

1. 不是所有请求都需要百万上下文

大部分日常对话用默认的200K就够了，只有需要分析大量代码或文档时才开1M。

2. Context Compaction（上下文压缩）

Anthropic在beta版里加了这个功能，当对话接近限制时会自动总结旧上下文，有效延长可用长度。

3. 批量处理更高效

如果需要分析多个文件，一次性打包发送比分多次询问更省Token（因为不用重复发送系统提示）。

我测试了大约20轮对话，总花费大概$2左右，算下来比雇一个实习生便宜多了（而且不用管饭）。

局限性和注意事项

当然，Sonnet 4.6也不是万能的，有几个地方需要注意：

1. 百万上下文目前还是beta

最大输出Token限制在64K，而且不是所有功能都稳定。生产环境用的话建议先测试。

2. 延迟问题

处理长上下文确实需要时间。我测试的8万Token请求，平均响应时间在20-40秒。如果是实时交互场景，可能需要优化一下体验。

3. 不是Token越多越好

有时候给太多无关内容反而会干扰模型的判断。建议只发送相关文件，或者先做个目录结构说明。

4. API稳定性

国内直接调Anthropic的API经常抽风，建议用可靠的中转服务（像我上面提到的KL-API），或者自己搭代理。

写在最后

用了两周Claude Sonnet 4.6，我的感受是：这玩意儿真的改变了我的工作流。

以前用AI辅助编程，本质上是"我有一个小任务，AI帮我写段代码"。现在是"我有一个复杂项目，AI帮我理解、分析、优化"。

从"代码生成器"到"代码理解者"，这个转变的意义可能比想象中更大。

如果你还没试过Sonnet 4.6的百万上下文，我建议找个自己的项目测试一下。那种"把整个代码库扔给AI然后问问题"的体验，真的很不一样。

最后，关于KL-API的使用，如果你有国内调用Claude系列模型的需求，可以尝试一下。毕竟工具是为人服务的，能解决问题才是王道。
